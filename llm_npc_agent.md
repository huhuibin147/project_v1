# LLM 驱动的 NPC 对话与决策 — 展开方案

## 核心理念

传统 NPC：**脚本驱动**，行为固定、对话预设、无记忆
LLM NPC：**智能驱动**，行为动态、对话开放、有记忆、有个性

```
传统 NPC：
  玩家靠近 → 触发脚本 → 播放预设对话 → 结束

LLM NPC：
  玩家靠近 → NPC 感知 → 调用 LLM 推理 → 生成对话/决策 → 记忆更新 → 动态响应
```

---

## 一、NPC 对话系统

### 1.1 基础对话引擎

NPC 能根据自身身份、与玩家的关系、当前情境进行自由对话。

**核心要素**：
- **角色设定（System Prompt）**：定义 NPC 的身份、背景、说话风格
- **上下文注入**：当前游戏状态、玩家信息、历史交互
- **输出控制**：限制回复长度、语气、格式

**示例 Prompt 结构**：
```
你是一个铁匠，名叫老王，性格豪爽，在这个村庄经营了 20 年。
当前村庄正遭受怪物袭击，你很担忧家人安全。
玩家是刚到村庄的冒险者。

玩家说：{玩家输入}

请用 1-2 句话回复，保持角色一致性。
```

### 1.2 多轮对话记忆

NPC 能记住与玩家的历史对话，实现连贯的多轮交互。

**实现方案**：
- **滑动窗口**：保留最近 N 轮对话
- **摘要压缩**：每 N 轮生成对话摘要，替代原始历史
- **关键信息提取**：提取玩家偏好、承诺、重要事件存入记忆

### 1.3 对话分支与意图识别

NPC 不只是聊天，还能根据玩家意图触发游戏行为。

**玩家意图分类**：
- 闲聊 → NPC 随意回应
- 询问任务 → NPC 描述可用任务
- 交易 → 打开商店界面
- 询问情报 → NPC 分享已知信息
- 侮辱 → NPC 降低好感度，可能拒绝服务

**实现**：LLM 输出结构化 JSON，包含 `intent` + `response` + `action`

## 二、NPC 决策系统

### 2.1 日程与行为规划

NPC 根据时间、事件、目标自主决定做什么。

**示例**：
```
早上 6:00  → 起床、开门营业
上午 10:00 → 锻造武器
中午 12:00 → 去酒馆吃饭
下午 14:00 → 与村民交谈（可能触发情报传递）
晚上 20:00 → 关门回家
发现怪物  → 拿起武器防御 / 逃跑 / 求助
```

**LLM 的作用**：当预设日程无法覆盖时（如突发事件），调用 LLM 进行实时决策。

### 2.2 目标驱动行为（GOAP + LLM）

NPC 有自己的目标，LLM 帮助规划达成目标的行动序列。

```
NPC 目标：保护村庄安全

LLM 规划：
1. 先检查武器库存
2. 如果武器不足 → 联系铁匠购买
3. 组织村民巡逻
4. 在村口设置陷阱
5. 如果怪物太强 → 向冒险者公会求援
```

### 2.3 情境感知决策

NPC 根据游戏世界的实时状态做出反应。

**感知输入**：
- 玩家行为（偷窃、攻击、帮助）
- 环境变化（天气、时间、战斗）
- 其他 NPC 状态（有人受伤、有人死亡）
- 世界事件（Boss 来袭、节日庆典）

**决策输出**：
- 情绪变化（害怕、高兴、愤怒）
- 行为改变（逃跑、跟随、攻击）
- 对话内容变化（语气、态度）

## 三、NPC 记忆系统

### 3.1 三层记忆架构

```
┌─────────────────────────────────────────┐
│           长期记忆 (Long-term)           │
│  世界观、人生经历、核心信念、重要关系      │
│  → 写入角色设定，始终存在于上下文中        │
├─────────────────────────────────────────┤
│           中期记忆 (Mid-term)            │
│  与玩家的交互历史、近期重要事件            │
│  → 摘要存储，检索时注入上下文              │
├─────────────────────────────────────────┤
│           短期记忆 (Short-term)          │
│  当前对话上下文、刚刚发生的事件            │
│  → 滑动窗口，直接作为对话历史              │
└─────────────────────────────────────────┘
```

### 3.2 记忆存储方案

- **向量数据库**（ChromaDB / FAISS）：存储记忆片段，语义检索
- **结构化存储**（JSON / SQLite）：存储关系、事件、状态
- **LLM 摘要**：定期将短期记忆压缩为中期记忆

### 3.3 记忆影响行为

```
记忆：玩家上周救过我的命
→ 对话：态度友好，愿意提供折扣
→ 决策：在玩家遇到危险时主动帮忙

记忆：玩家偷过我的东西
→ 对话：态度冷淡，言语中带有警惕
→ 决策：拒绝赊账，高价出售
```

## 四、NPC 个性与情感系统

### 4.1 性格模型

用简单的性格参数影响 LLM 的输出风格。

**参数示例**：
```json
{
  "name": "老王",
  "personality": {
    "friendliness": 0.8,
    "courage": 0.6,
    "greed": 0.3,
    "humor": 0.7
  },
  "current_mood": "担忧",
  "relationship_with_player": 0.5
}
```

**注入 Prompt**：
```
你是一个友善（friendliness=0.8）、有点幽默（humor=0.7）的铁匠。
你对面前的冒险者好感度为 0.5（中立偏友好）。
你当前的情绪是"担忧"，因为村庄正遭受威胁。
```

### 4.2 情感状态机

NPC 的情绪会随事件动态变化。

```
事件：玩家完成任务
→ 好感度 +0.2，情绪 → 感激
→ 对话变得更热情

事件：玩家攻击村民
→ 好感度 -0.5，情绪 → 愤怒/恐惧
→ 对话变得敌对，可能呼叫守卫

事件：村庄被袭击
→ 情绪 → 恐惧
→ 行为 → 寻找掩护 / 拿起武器
```

### 4.3 说话风格一致性

不同 NPC 有不同说话方式，通过 Prompt 约束。

| NPC 类型 | 说话风格示例 |
|---------|-------------|
| 铁匠 | 粗犷直接，"有啥需要的？俺这刚出炉一批好剑！" |
| 学者 | 文雅谨慎，"此事需从长计议，容我查阅古籍..." |
| 盗贼 | 隐晦暗示，"嘿嘿，有些事...不方便明说，你懂的。" |
| 贵族 | 高傲礼貌，"阁下光临寒舍，不知有何贵干？" |

## 五、多 NPC 协作与社交

### 5.1 NPC 间信息传递

NPC 之间可以交流信息，形成"传闻系统"。

```
玩家在酒馆杀了人 → 酒馆老板看到
→ 酒馆老板告诉邻居
→ 邻居告诉守卫
→ 守卫开始追捕玩家

信息传播通过 LLM 调用实现，每个 NPC 根据自己的性格决定是否传播、如何传播。
```

### 5.2 NPC 群体决策

面对重大事件，多个 NPC 共同讨论并做出决策。

```
事件：怪物大军来袭

NPC 会议（LLM 多角色模拟）：
村长：我们应该组织防御！
铁匠：我有足够的武器，但人手不够。
商人：要不我们花钱雇佣冒险者？
村长：好主意，但钱从哪来？
商人：我可以出一半，但事后要减税。
村长：成交。

→ 决策结果：集资雇佣冒险者
```

### 5.3 NPC 关系网络

NPC 之间有亲疏关系，影响互动。

```
老王（铁匠）→ 好友 → 小李（猎人）
老王 → 敌对 → 张三（竞争对手铁匠）
老王 → 恋人 → 小红（酒馆老板娘）

玩家攻击小李 → 老王对玩家好感度大幅下降
玩家帮老王打败张三 → 老王对玩家好感度大幅提升
```

### 6.2 NPC Agent 类设计

```python
class NPCAgent:
    def __init__(self, name, personality, backstory):
        self.name = name
        self.personality = personality      # 性格参数
        self.backstory = backstory          # 背景故事
        self.memory = MemorySystem()        # 记忆系统
        self.mood = "neutral"               # 当前情绪
        self.relationships = {}             # 与其他角色的关系

    def perceive(self, event):
        """感知游戏世界事件"""
        self.memory.add_short_term(event)
        self.update_mood(event)

    def decide(self, context):
        """根据情境做出决策"""
        prompt = self.build_decision_prompt(context)
        return llm_call(prompt)

    def speak(self, player_input, context):
        """生成对话回复"""
        prompt = self.build_dialogue_prompt(player_input, context)
        response = llm_call(prompt)
        self.memory.add_short_term({"role": "player", "content": player_input})
        self.memory.add_short_term({"role": "self", "content": response})
        return response

    def plan(self, goal):
        """规划达成目标的行动序列"""
        prompt = self.build_planning_prompt(goal)
        return llm_call(prompt)
```

## 九、关键技术难点与解决思路

### 难点 1：LLM 响应延迟
- **问题**：NPC 对话等待时间长，破坏沉浸感
- **方案**：预生成常用回复缓存；异步调用；显示 NPC 思考动画

### 难点 2：角色一致性
- **问题**：LLM 可能"出戏"，说出不符合角色的话
- **方案**：强约束 System Prompt；输出格式校验；后处理过滤

### 难点 3：Token 成本
- **问题**：每个 NPC 每次交互都调用 LLM，成本高
- **方案**：本地小模型处理简单对话；仅复杂场景调用大模型；对话摘要压缩上下文

### 难点 4：游戏逻辑耦合
- **问题**：LLM 输出的"行动"需要转化为游戏内的实际操作
- **方案**：结构化输出（JSON）；定义有限的 action 集合；LLM 只做决策，执行由游戏引擎负责
