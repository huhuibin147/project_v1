# 现有模块优化方向分析

> 生成日期：2026-05-12
> 基于版本：锻造与词条系统完成后

---

## 一、后端模块

### 1.1 main.py（API 路由层）

**当前状态**：单文件 1100+ 行，包含 46 个路由端点

| 问题 | 优先级 | 说明 |
|------|--------|------|
| 文件过长，职责混杂 | 高 | 路由、业务逻辑、数据模型定义全部集中在一个文件，维护困难 |
| 全局状态管理 | 高 | `npc_agents` 字典、`player_profile` 单例、`_combat_sessions` 均为全局变量，多用户场景下会冲突 |
| 重复的 Pydantic 模型 | 中 | 每个路由单独定义 Request 模型，可统一归类 |
| 缺少错误处理中间件 | 中 | 无统一的异常捕获和错误返回格式 |
| 缺少 API 版本控制 | 低 | 未来接口变更时可能破坏前端兼容性 |

**优化建议**：
- 按功能模块拆分路由（`/api/combat/*` → `routes/combat.py`，`/api/forge/*` → `routes/forge.py`）
- 使用 FastAPI 的 `APIRouter` 组织路由
- 引入数据库层替代文件存档，支持多用户并发
- 添加统一的异常处理中间件和日志记录

### 1.2 combat_engine.py（战斗引擎）

**当前状态**：约 1050 行，包含伤害计算、状态效果、天赋触发、词条触发、技能执行、怪物 AI

| 问题 | 优先级 | 说明 |
|------|--------|------|
| 单敌战斗 | 高 | `CombatSession` 只支持单个怪物，设计文档中规划的多敌人（1-3 个）和 BOSS 战未实现 |
| 伤害公式简单 | 中 | 当前公式 `attack * 100/(100+defense)` 缺乏属性克制、装备特效加成等维度 |
| 词条触发逻辑分散 | 中 | `_apply_affix_*` 函数分散在文件中，新增词条类型需要修改多处 |
| 状态效果硬编码 | 中 | `_process_effects` 中每个效果类型用 `if/elif` 分支处理，扩展新效果需修改核心函数 |
| 技能系统耦合 | 低 | `_execute_skill` 中直接处理所有技能类型，技能增多后难以维护 |
| 战斗快照不完整 | 低 | 创建战斗会话时只保存基础属性，未保存完整玩家状态 |

**优化建议**：
- 实现多敌人战斗：将 `monster_hp`/`monster_effects` 改为怪物列表
- 引入策略模式处理状态效果：每个效果类型注册为独立处理器
- 词条触发改为事件驱动：定义 `on_attack`、`on_hit` 等事件，词条注册监听器
- 伤害公式增加属性克制系数（火>草>水>火）

### 1.3 player_profile.py（玩家档案）

**当前状态**：约 650 行，包含存档管理、装备系统、属性计算

| 问题 | 优先级 | 说明 |
|------|--------|------|
| 文件存档 | 高 | 使用 JSON 文件存储，无并发保护，多用户场景下数据可能损坏 |
| `_recalc_stats` 频繁调用 | 中 | 每次装备变更都触发全量重算并保存，性能浪费 |
| 装备词条查询效率 | 中 | `get_item_affixes` 每次遍历整个背包查找匹配物品，背包大时效率低 |
| 属性计算链路长 | 中 | 基础属性 → 等级加成 → 装备加成 → 词条加成 → 天赋加成，链路长且耦合 |
| 缺少属性上限校验 | 低 | 极端情况下属性可能溢出或为负 |

**优化建议**：
- 引入缓存机制：装备未变更时复用上次计算结果
- 属性计算改为增量更新：只重新计算变更的部分
- 物品实例使用唯一 ID 索引，避免遍历查找

### 1.4 forge_system.py & affix_system.py（锻造与词条）

**当前状态**：锻造系统 25 种配方，词条系统 30+ 词条

| 问题 | 优先级 | 说明 |
|------|--------|------|
| 词条权重固定 | 中 | `affixes.json` 中权重写死，无法根据玩家等级动态调整 |
| 锻造无保底机制 | 中 | 连续失败无补偿，玩家体验可能不佳 |
| 缺少词条洗练 | 中 | 无法替换或重roll装备上的词条 |
| 配方无解锁进度 | 低 | 所有配方一开始就可见，缺少探索感 |

**优化建议**：
- 增加锻造保底：连续 N 次失败后下次必成功
- 实现词条洗练功能（消耗金币重roll词条）
- 配方解锁与地图进度挂钩

### 1.5 npc_agent.py（NPC 对话）

**当前状态**：依赖 LLM API 进行意图识别和对话生成

| 问题 | 优先级 | 说明 |
|------|--------|------|
| LLM 调用成本高 | 高 | 每次对话都调用 LLM API，响应慢且有费用 |
| 意图识别不稳定 | 高 | 依赖 LLM 输出 JSON 格式，可能因模型输出格式问题导致解析失败 |
| 记忆系统简单 | 中 | 只保留最近 10 条对话历史，缺少长期记忆和关键事件记录 |
| 好感度影响有限 | 中 | 好感度变化只影响 mood 字段，未深度影响 NPC 行为 |
| 无对话缓存 | 低 | 相同问题的回答每次都重新生成 |

**优化建议**：
- 引入本地意图分类器（规则 + 关键词匹配）替代部分 LLM 调用
- 实现分层记忆：短期记忆（最近对话）+ 长期记忆（关键事件）
- 好感度影响商店折扣、任务奖励、NPC 主动行为
- 添加常见问题的回答缓存

### 1.6 item_system.py（物品系统）

**当前状态**：物品数据库 + 基础背包操作

| 问题 | 优先级 | 说明 |
|------|--------|------|
| 物品效果硬编码 | 中 | `ITEM_EFFECTS` 字典写死在代码中，新增物品需修改代码 |
| 缺少物品分类 | 低 | 所有物品平铺在一个 JSON 中，缺少分类管理 |
| 物品堆叠逻辑简单 | 低 | 只按 `item_id` 堆叠，未区分实例化物品（带词条的装备） |

**优化建议**：
- 将物品效果移到 `items.json` 配置中
- 区分可堆叠物品和不可堆叠物品（装备类不堆叠）

### 1.7 quest_manager.py（任务系统）

| 问题 | 优先级 | 说明 |
|------|--------|------|
| 任务进度检查手动触发 | 中 | 需要前端主动调用 `/api/quests/progress` 更新进度 |
| 缺少任务链 | 低 | 任务之间无关联，无法形成任务线 |
| 每日任务重置逻辑简单 | 低 | 基于日期字符串比较，跨时区可能有问题 |

## 二、前端模块

### 2.1 game.js（游戏主循环）

**当前状态**：主循环 + 画布渲染 + 输入处理

| 问题 | 优先级 | 说明 |
|------|--------|------|
| 全局变量过多 | 高 | 大量 `let` 全局变量，命名空间污染严重 |
| 无模块化 | 高 | 所有逻辑在全局作用域，难以维护和测试 |
| 渲染性能 | 中 | 每帧全量重绘，未使用脏矩形优化 |
| 输入处理耦合 | 中 | 键盘事件直接绑定到全局函数，缺少输入管理器 |
| 缺少加载状态 | 低 | 资源加载时无 Loading 提示 |

**优化建议**：
- 引入模块模式或 ES6 模块，封装各子系统
- 实现输入管理器，统一处理键盘/触摸事件
- 添加脏矩形渲染优化，减少不必要的重绘
- 使用 `requestIdleCallback` 处理非紧急更新

### 2.2 combat.js（战斗 UI）

| 问题 | 优先级 | 说明 |
|------|--------|------|
| 战斗日志渲染性能 | 中 | 每次更新全量重绘日志区域，日志多时卡顿 |
| 缺少动画过渡 | 中 | 伤害数字、状态效果无动画效果 |
| 战斗操作按钮分散 | 低 | 攻击/防御/技能/物品/逃跑通过键盘数字键，缺少可视化按钮 |

**优化建议**：
- 战斗日志使用虚拟滚动，只渲染可见区域
- 添加伤害数字弹出动画
- 增加移动端友好的战斗操作按钮

### 2.3 inventory.js（背包系统）

| 问题 | 优先级 | 说明 |
|------|--------|------|
| 装备卡片渲染 | 中 | 背包物品多时渲染慢，未做分页或虚拟列表 |
| 词条显示空间有限 | 中 | 装备卡片上词条信息只显示名称，描述需要 hover |
| 缺少拖拽操作 | 低 | 无法拖拽装备到装备槽位 |

### 2.4 forge.js（锻造面板）

| 问题 | 优先级 | 说明 |
|------|--------|------|
| 无锻造动画 | 中 | 锻造过程瞬间完成，缺少进度反馈 |
| 配方列表无搜索 | 低 | 配方多时查找不便 |

### 2.5 npc.js & dialogue.js（NPC 与对话）

| 问题 | 优先级 | 说明 |
|------|--------|------|
| 对话气泡样式简单 | 中 | 纯文本显示，缺少气泡动画和打字机效果 |
| LLM 响应等待无反馈 | 中 | 调用 LLM 时无 Loading 提示，用户可能重复发送 |
| 对话历史查看不便 | 低 | 需要打开专门面板查看历史 |

### 2.6 map.js & player.js（地图与角色）

| 问题 | 优先级 | 说明 |
|------|--------|------|
| 地图加载性能 | 中 | 大地图（80×60）初始化时遍历所有瓦片，可能卡顿 |
| 角色动画简单 | 中 | 只有 4 方向行走帧，缺少攻击、受击等动画 |
| 摄像机跟随生硬 | 低 | 摄像机直接跳到目标位置，缺少平滑过渡 |

## 四、架构层面

### 4.1 前后端通信

| 问题 | 优先级 | 说明 |
|------|--------|------|
| 无 WebSocket | 高 | 战斗、对话等实时交互使用 HTTP 轮询，延迟高 |
| 缺少请求合并 | 中 | 多个 API 调用独立发送，可增加批量接口 |
| 无请求重试机制 | 中 | 网络异常时无自动重试 |

### 4.2 存档系统

| 问题 | 优先级 | 说明 |
|------|--------|------|
| 文件存档无加密 | 高 | 存档文件明文存储，用户可手动修改 |
| 无自动备份 | 中 | 存档损坏时无法恢复 |
| 存档体积增长 | 低 | 对话历史、任务记录随时间增长 |

### 4.3 性能

| 问题 | 优先级 | 说明 |
|------|--------|------|
| 全局 DB 加载 | 中 | 启动时加载所有 JSON 配置到内存，冷启动慢 |
| 无缓存策略 | 中 | 物品信息、NPC 配置等静态数据每次都从文件读取 |
| 前端无资源压缩 | 低 | JS/CSS 未压缩，加载时间长 |

## 五、优先级排序

### P0（必须优化）

1. **main.py 模块化拆分** — 使用 APIRouter 按功能拆分路由
2. **多敌人战斗系统** — 实现设计文档中的 1-3 个怪物 + BOSS 战
3. **LLM 意图识别优化** — 引入本地规则匹配，降低 API 调用频率
4. **存档安全性** — 添加存档校验和/加密

### P1（建议优化）

5. **前端模块化** — 消除全局变量，引入模块封装
6. **状态效果策略模式** — 替换 if/elif 分支为注册式处理器
7. **词条事件驱动** — 词条触发改为事件监听模式
8. **锻造保底机制** — 连续失败补偿
9. **战斗日志虚拟滚动** — 提升大量日志时的渲染性能
10. **属性计算缓存** — 减少重复计算

### P2（可选优化）

11. **WebSocket 实时通信** — 替代 HTTP 轮询
12. **对话缓存** — 常见问题回答缓存
13. **地图懒加载** — 大地图按需加载瓦片
14. **物品标签系统** — 按标签分类筛选
15. **怪物 AI 增强** — 更复杂的战斗行为

| 债务 | 影响 | 修复成本 |
|------|------|----------|
| `combat_engine.py` 1050 行单文件 | 维护困难，新增功能易引入 bug | 中 |
| `main.py` 1100 行单文件 | 同上 | 中 |
| 全局变量污染（前端） | 命名冲突风险，难以测试 | 高 |
| 文件存档无并发保护 | 多用户数据损坏 | 中 |
| `ITEM_EFFECTS` 硬编码 | 新增物品需改代码 | 低 |
| 伤害公式缺乏扩展性 | 新增属性类型需改核心逻辑 | 中 |
| 词条效果部分未实现 | 配置与代码不一致 | 低 |
