## 一、后端模块

### 1.1 main.py（API 路由层）

**当前状态**：已拆分为模块化路由结构，main.py 仅作为路由注册入口（约 135 行）

**已完成优化**：
- 按功能模块拆分路由到 `backend/routes/` 目录
- 使用 FastAPI 的 `APIRouter` 组织路由
- 添加统一的异常处理中间件（`validation_exception_handler`、`general_exception_handler`）
- Pydantic 模型统一归类到 `routes/models.py`
- 共享上下文通过 `routes/context.py` 管理

**路由模块结构**：
```
backend/routes/
├── __init__.py
├── context.py      # 共享应用上下文（player_profile, npc_agents, quest_manager）
├── models.py       # Pydantic 请求/响应模型
├── npc.py          # NPC 相关路由（聊天、商店、交易、服务）
├── player.py       # 玩家相关路由（档案、背包、装备、存档）
├── map.py          # 地图相关路由（瓦片、地图数据、传送、物件交互）
├── combat.py       # 战斗相关路由（开始、行动、结束）
├── forge.py        # 锻造/词条相关路由
└── quest.py        # 任务/天赋相关路由
```

**当前问题**：

| 问题 | 优先级 | 说明 |
|------|--------|------|
| 全局状态管理 | 中 | `npc_agents` 字典、`player_profile` 单例、`_combat_sessions` 均为全局变量，模块间耦合紧密 |
| 缺少 API 版本控制 | 低 | 未来接口变更时可能破坏前端兼容性 |
| 缺少请求日志 | 低 | 无统一的请求日志记录中间件 |

**后续优化建议**：
- 引入依赖注入模式替代全局状态（FastAPI 的 `Depends`）
- 添加请求日志中间件
- 添加 API 版本前缀（如 `/api/v1/`）

### 1.2 combat_engine.py（战斗引擎）

**当前状态**：已拆分为模块化结构，原 combat_engine.py 保留作为兼容层，新功能使用 `backend/combat/` 模块

**已完成优化**：
- 按功能拆分为独立模块：`session.py`、`damage.py`、`effects.py`、`events.py`、`monster_ai.py`、`skills.py`、`turn.py`
- 使用策略模式处理状态效果：每种效果类型注册为独立 `EffectHandler`
- 事件驱动系统：词条和天赋通过 `EventDispatcher` 注册监听器触发
- 伤害公式增加属性克制系数（火>草>水>火，克制 1.5x，被克制 0.67x）
- 创建 `engine.py` 兼容层，保持与现有路由代码的兼容性
- 更新 `routes/combat.py` 使用新模块

**新模块结构**：
```
backend/combat/
├── __init__.py
├── session.py          # CombatSession 类 + 会话管理
├── damage.py           # 伤害计算（含属性克制）
├── effects.py          # 状态效果系统（策略模式）
├── events.py           # 事件驱动系统（词条/天赋触发）
├── monster_ai.py       # 怪物 AI 决策
├── skills.py           # 技能执行
├── turn.py             # 回合解析核心
└── engine.py           # 对外暴露的统一接口
```

**当前问题**：

| 问题 | 优先级 | 说明 |
|------|--------|------|
| 属性克制未接入战斗 | 高 | `calc_damage` 已支持 `attacker_element`/`defender_element` 参数（火>草>水>火，克制 1.5x，被克制 0.67x），但 `resolve_turn`、`execute_skill`、`execute_action` 中均未传入元素属性，属性克制系统形同虚设 |
| 事件系统每次重建 | 中 | `EventDispatcher` 在每次回合解析时 `clear_dispatcher()` + 重新注册，应优化为会话级缓存，仅在会话创建时注册一次 |
| 怪物技能单一 | 中 | 怪物特殊技能仅支持 `apply_effect` 一种类型，缺少 AOE 攻击、召唤小怪、治疗、护盾、属性攻击等多样化技能 |
| 战斗快照不完整 | 低 | 创建战斗会话时只保存基础属性，未保存完整玩家状态（如装备详情、背包物品引用） |
| 缺少战斗难度动态调整 | 低 | 怪物属性固定，未根据玩家等级/装备动态调整，后期低级怪物无挑战性 |
| 技能冷却全局共享 | 低 | `skill_cooldowns` 在 `CombatSession` 级别管理，同一技能的所有等级共享冷却 |

**已完成优化**：
- ✅ 多敌人战斗：`CombatSession` 支持 `monsters: list[MonsterInstance]`，支持 1-3 个怪物 + BOSS 战
- ✅ 怪物组系统：地图配置支持 `monster_groups`，可配置多怪物编队
- ✅ BOSS 阶段系统：`MonsterInstance` 支持 `phases` 配置，HP 阈值触发阶段切换，阶段可改变 AI 行为和属性
- ✅ BOSS 免疫机制：BOSS 免疫眩晕/冻结，精英 BOSS 额外免疫中毒
- ✅ AOE 技能：`execute_skill` 支持 `aoe` 属性，群体伤害按目标数量递减（1 目标 100% → 3 目标 65%）
- ✅ 目标选择：前端支持多怪物槽位，Tab 键切换目标，点击选择目标
- ✅ 状态效果互斥：灼烧和冻结互斥，施加一个会移除另一个
- ✅ 可堆叠效果：流血效果支持叠加（最多 3 层），每层增加伤害

**后续优化建议**：
- 在 `resolve_turn`、`execute_skill`、`execute_action` 中传入元素属性参数，激活属性克制系统
- 优化事件系统：监听器在 `create_combat_session` 时注册一次，`remove_session` 时清理
- 扩展怪物特殊技能类型：AOE 攻击、召唤、治疗、护盾、属性攻击、debuff 组合
- 添加战斗难度缩放：根据玩家等级动态调整怪物属性（如 `monster_stats *= 1 + (player_level - monster_level) * 0.1`）
- 完善战斗快照：保存装备词条详情、背包可用物品列表，支持战斗中使用物品的完整校验

### 1.3 player_profile.py（玩家档案）

**当前状态**：约 700 行，包含存档管理、装备系统、属性计算

**已完成优化**：
- 存档备份与恢复：保存前自动备份为 `.json.bak`，加载失败时尝试从备份恢复
- 属性计算缓存：使用 `StatsCache` 类，装备/天赋/等级未变更时复用计算结果
- 物品索引优化：使用 `_inventory_index` 字典，查询复杂度从 O(N) 降为 O(1)
- 属性计算模块化：拆分为 `_calculate_stats()` 和 `_apply_stats()` 两个阶段
- 属性上限校验：所有属性限制在 [1, 99999] 范围内，防止溢出或为负
- 数据序列化拆分：新增 `_to_dict()` 和 `_from_dict()` 方法，职责更清晰

**剩余问题**：
| 问题 | 优先级 | 说明 |
|------|--------|------|
| 装备词条查询效率 | 低 | `get_item_affixes` 仍遍历背包，但调用次数因缓存大幅减少 |

**新增类/方法**：
- `StatsCache`：属性计算缓存类，使用 MD5 哈希判断是否需要重算
- `_backup_save()` / `_restore_from_backup()`：存档备份与恢复
- `_verify_save()`：存档完整性校验
- `_to_dict()` / `_from_dict()`：数据序列化/反序列化
- `_rebuild_inventory_index()`：重建物品索引
- `_calculate_stats()` / `_apply_stats()`：属性计算与应用的分离

|------|--------|------|
| 配方无解锁进度 | 低 | 所有配方一开始就可见，缺少探索感 |

- `execute_forge()` 新增 `forge_streaks` 参数，返回结果包含更新后的 streaks

- `_get_dynamic_weight()` / `_level_in_range()`：动态权重计算
- `forge_streaks`：玩家存档字段，记录各配方连续失败次数

### 1.5 npc_agent.py（NPC 对话）
