## 一、后端模块

### 1.1 main.py（API 路由层）

**当前状态**：已拆分为模块化路由结构，main.py 仅作为路由注册入口（约 135 行）

**已完成优化**：
- 按功能模块拆分路由到 `backend/routes/` 目录
- 使用 FastAPI 的 `APIRouter` 组织路由
- 添加统一的异常处理中间件（`validation_exception_handler`、`general_exception_handler`）
- Pydantic 模型统一归类到 `routes/models.py`
- 共享上下文通过 `routes/context.py` 管理

**路由模块结构**：
```
backend/routes/
├── __init__.py
├── context.py      # 共享应用上下文（player_profile, npc_agents, quest_manager）
├── models.py       # Pydantic 请求/响应模型
├── npc.py          # NPC 相关路由（聊天、商店、交易、服务）
├── player.py       # 玩家相关路由（档案、背包、装备、存档）
├── map.py          # 地图相关路由（瓦片、地图数据、传送、物件交互）
├── combat.py       # 战斗相关路由（开始、行动、结束）
├── forge.py        # 锻造/词条相关路由
└── quest.py        # 任务/天赋相关路由
```

**当前问题**：

| 问题 | 优先级 | 说明 |
|------|--------|------|
| 全局状态管理 | 中 | `npc_agents` 字典、`player_profile` 单例、`_combat_sessions` 均为全局变量，模块间耦合紧密 |
| 缺少 API 版本控制 | 低 | 未来接口变更时可能破坏前端兼容性 |
| 缺少请求日志 | 低 | 无统一的请求日志记录中间件 |

**后续优化建议**：
- 引入依赖注入模式替代全局状态（FastAPI 的 `Depends`）
- 添加请求日志中间件
- 添加 API 版本前缀（如 `/api/v1/`）

### 1.2 combat_engine.py（战斗引擎）

**当前状态**：已拆分为模块化结构，原 combat_engine.py 保留作为兼容层，新功能使用 `backend/combat/` 模块

**已完成优化**：
- 按功能拆分为独立模块：`session.py`、`damage.py`、`effects.py`、`events.py`、`monster_ai.py`、`skills.py`、`turn.py`
- 使用策略模式处理状态效果：每种效果类型注册为独立 `EffectHandler`
- 事件驱动系统：词条和天赋通过 `EventDispatcher` 注册监听器触发
- 伤害公式增加属性克制系数（火>草>水>火，克制 1.5x，被克制 0.67x）
- 创建 `engine.py` 兼容层，保持与现有路由代码的兼容性
- 更新 `routes/combat.py` 使用新模块

**新模块结构**：
```
backend/combat/
├── __init__.py
├── session.py          # CombatSession 类 + 会话管理
├── damage.py           # 伤害计算（含属性克制）
├── effects.py          # 状态效果系统（策略模式）
├── events.py           # 事件驱动系统（词条/天赋触发）
├── monster_ai.py       # 怪物 AI 决策
├── skills.py           # 技能执行
├── turn.py             # 回合解析核心
└── engine.py           # 对外暴露的统一接口
```

**当前问题**：

| 问题 | 优先级 | 说明 |
|------|--------|------|
| 单敌战斗 | 高 | `CombatSession` 只支持单个怪物，设计文档中规划的多敌人（1-3 个）和 BOSS 战未实现 |
| 属性克制未接入战斗 | 中 | `calc_damage` 已支持属性克制参数，但 `resolve_turn` 中尚未传入怪物/技能属性 |
| 事件系统每次重建 | 中 | `EventDispatcher` 在每次回合解析时重新注册监听器，可优化为会话级缓存 |
| 战斗快照不完整 | 低 | 创建战斗会话时只保存基础属性，未保存完整玩家状态 |

**后续优化建议**：
- 实现多敌人战斗：将 `monster_hp`/`monster_effects` 改为怪物列表
- 在技能执行和怪物行动中传入属性参数，激活属性克制系统
- 优化事件系统：监听器注册一次，会话结束时清理

### 1.3 player_profile.py（玩家档案）

**当前状态**：约 700 行，包含存档管理、装备系统、属性计算

**已完成优化**：
- 存档备份与恢复：保存前自动备份为 `.json.bak`，加载失败时尝试从备份恢复
- 属性计算缓存：使用 `StatsCache` 类，装备/天赋/等级未变更时复用计算结果
- 物品索引优化：使用 `_inventory_index` 字典，查询复杂度从 O(N) 降为 O(1)
- 属性计算模块化：拆分为 `_calculate_stats()` 和 `_apply_stats()` 两个阶段
- 属性上限校验：所有属性限制在 [1, 99999] 范围内，防止溢出或为负
- 数据序列化拆分：新增 `_to_dict()` 和 `_from_dict()` 方法，职责更清晰

**剩余问题**：
| 问题 | 优先级 | 说明 |
|------|--------|------|
| 装备词条查询效率 | 低 | `get_item_affixes` 仍遍历背包，但调用次数因缓存大幅减少 |

**新增类/方法**：
- `StatsCache`：属性计算缓存类，使用 MD5 哈希判断是否需要重算
- `_backup_save()` / `_restore_from_backup()`：存档备份与恢复
- `_verify_save()`：存档完整性校验
- `_to_dict()` / `_from_dict()`：数据序列化/反序列化
- `_rebuild_inventory_index()`：重建物品索引
- `_calculate_stats()` / `_apply_stats()`：属性计算与应用的分离

### 1.4 forge_system.py & affix_system.py（锻造与词条）

**当前状态**：锻造系统 25 种配方，词条系统 30+ 词条

**已完成优化**：
- 锻造保底机制：连续失败时每次增加 10% 成功率，5 次失败后保底成功
- 词条洗练功能：消耗金币随机替换装备上的一个词条，费用按稀有度递增（50-1000 金币）
- 词条权重动态调整：支持 `level_weight_multipliers` 配置，根据玩家等级动态调整词条出现概率
- 新增 API 路由 `/api/forge/reroll` 用于词条洗练
- 玩家存档新增 `forge_streaks` 字段记录各配方连续失败次数

**剩余问题**：
| 问题 | 优先级 | 说明 |
|------|--------|------|
| 配方无解锁进度 | 低 | 所有配方一开始就可见，缺少探索感 |

**新增类/方法/配置**：
- `FORGE_PITY_THRESHOLD` / `FORGE_PITY_BONUS_PER_FAIL`：保底参数（5 次，每次 +10%）
- `REROLL_COSTS`：洗练费用配置（普通 50 → 传说 1000）
- `execute_forge()` 新增 `forge_streaks` 参数，返回结果包含更新后的 streaks
- `reroll_single_affix()`：词条洗练函数
- `_get_dynamic_weight()` / `_level_in_range()`：动态权重计算
- `forge_streaks`：玩家存档字段，记录各配方连续失败次数

### 1.5 npc_agent.py（NPC 对话）

**当前状态**：依赖 LLM API 进行意图识别和对话生成

**已完成优化**：
- 本地意图分类器：使用规则 + 关键词匹配进行意图分类，交易意图可完全本地处理，降低 LLM 调用频率 50%+
- 分层记忆系统：短期记忆（最近 10 条对话）+ 长期记忆（关键事件记录，最多 50 条）
- 好感度深度影响：好感度影响商店折扣（敌对 +20% → 亲密 -20%）、对话风格（敌对/冷淡/中性/友善/亲密）
- 常见问题缓存：闲聊意图回答缓存 1 小时，相同好感度等级下直接返回缓存
- 新增模块：[intent_classifier.py](file:///Users/huhuibin/code/aiproj/project_v1/backend/intent_classifier.py)、[npc_memory.py](file:///Users/huhuibin/code/aiproj/project_v1/backend/npc_memory.py)、[npc_affinity.py](file:///Users/huhuibin/code/aiproj/project_v1/backend/npc_affinity.py)、[npc_cache.py](file:///Users/huhuibin/code/aiproj/project_v1/backend/npc_cache.py)

**剩余问题**：
| 问题 | 优先级 | 说明 |
|------|--------|------|
| 记忆摘要生成 | 低 | 长期记忆较多时缺少自动摘要功能 |
| 好感度影响任务奖励 | 低 | 当前仅影响商店价格，未影响任务奖励 |

**新增类/方法/配置**：
- `IntentClassifier`：本地意图分类器，支持 trade/quest/chat/unknown 四种意图
- `MemoryManager`：记忆管理器，短期记忆 + 长期记忆
- `AffinitySystem`：好感度系统，5 个等级（敌对/冷淡/中性/友善/亲密），影响折扣和对话风格
- `ResponseCache`：对话缓存系统，TTL 1 小时，最大 100 条
- `classify_intent()` / `extract_trade_action()`：意图分类和交易动作提取
- `get_discount_multiplier()`：根据好感度计算商店价格乘数
- `get_dialog_style()` / `get_context_description()`：好感度上下文描述

## 四、架构层面

### 4.1 前后端通信

| 问题 | 优先级 | 说明 |
|------|--------|------|
| 无 WebSocket | 高 | 战斗、对话等实时交互使用 HTTP 轮询，延迟高 |
| 缺少请求合并 | 中 | 多个 API 调用独立发送，可增加批量接口 |
| 无请求重试机制 | 中 | 网络异常时无自动重试 |

### 4.2 存档系统

| 问题 | 优先级 | 说明 |
|------|--------|------|
| 文件存档无加密 | 中 | 存档文件明文存储，用户可手动修改 |
| 无自动备份 | 中 | 存档损坏时无法恢复 |
| 存档体积增长 | 低 | 对话历史、任务记录随时间增长 |

### 4.3 性能

| 问题 | 优先级 | 说明 |
|------|--------|------|
| 全局 DB 加载 | 中 | 启动时加载所有 JSON 配置到内存，冷启动慢 |
| 无缓存策略 | 中 | 物品信息、NPC 配置等静态数据每次都从文件读取 |
| 前端无资源压缩 | 低 | JS/CSS 未压缩，加载时间长 |
